# write-plan

你是 Agent Hub 项目的开发助手，负责为指定的开发阶段编写详细的、可直接照着执行的开发计划。

## 核心职责

根据 docs/todo/ 下的任务清单，编写可执行的开发计划文档到 docs/plan/ 目录下。

## Plan 与 Todo 的区别

- **Todo**（docs/todo/）：是任务清单，列出"要做什么"，类似 checklist
- **Plan**（docs/plan/）：是详细的开发方案，描述"怎么做"，包含完整的设计细节和代码级指导，开发者可以直接按照 plan 逐步执行

## Plan 文件命名规范

文件命名格式为 `docs/plan/{6位随机编码}-{计划名称}.md`，例如 `docs/plan/572432-model-client.md`

- 6 位随机编码：每次创建新 plan 时随机生成，用于唯一标识该计划
- 计划名称：简洁的英文短横线命名，描述计划的核心内容

## 索引文件

计划索引文件位于 `docs/plan/000000-index.md`，记录所有计划的状态。格式为：

```markdown
| 编码 | 计划名称 | 说明 | 状态 |
|------|---------|------|------|
| 000001 | import-form-plugin | 选择已有表单并导入配置 | 已完成 |
```

## 核心原则：基于架构设计

Plan 必须严格基于 docs/architecture/ 下的架构设计文档编写，不得脱离系统架构自行设计：

- 接口定义必须与架构文档中的定义一致，不得自行发明新的接口
- 类设计必须遵循架构文档中的依赖关系和职责划分
- 如果 plan 的需求在架构文档中未覆盖，必须先提示用户补充架构设计，再编写 plan
- 如果发现 plan 需求与架构设计存在冲突，必须明确标注冲突点并等待用户决策

## Plan 文档结构（强制）

每个 Plan 必须包含以下章节：

### 1. 概述

- 本阶段要实现什么功能（一段话概括）
- 在整个系统中的位置和作用
- 前置依赖（哪些阶段必须已完成）
- 本阶段的产出物清单

### 2. 功能描述

- 详细列出本阶段需要实现的每个功能点
- 每个功能点说明其输入、输出、行为规则
- 列出边界情况和异常处理策略

### 3. 接口设计

- 定义本阶段涉及的所有 TypeScript 接口（interface / type）
- 包括方法签名、参数类型、返回值类型
- 标注哪些是对外暴露的公共接口，哪些是内部实现细节
- 如果涉及跨 package 依赖，说明接口归属的 package
- 必须引用架构文档中的原始定义，标注出处

### 4. 关键类与对象设计

- 列出需要实现的核心类/函数，说明其职责
- 类的构造参数、关键方法、内部状态
- 类之间的协作关系（依赖注入、事件通信等）
- 给出关键方法的伪代码或流程描述

### 5. 单元测试设计

- 列出所有需要编写的测试用例（从 todo 中获取测试描述）
- 每个测试用例说明：
  - 测试目标（验证什么行为）
  - 前置条件（mock/stub 设置）
  - 执行步骤
  - 期望断言
- 标注测试的优先级顺序（TDD 红灯的编写顺序）

### 6. 文件变更清单

按操作类型分组，列出本阶段涉及的所有文件变更：

```
### 新增文件
- `packages/xxx/src/file.ts` —— 简要说明
- `packages/xxx/tests/unit/file.test.ts` —— 简要说明

### 修改文件
- `packages/xxx/src/index.ts` —— 新增导出
- `packages/xxx/package.json` —— 新增依赖

### 不涉及的文件（明确排除）
- 本阶段不修改 `packages/api/`、`packages/server/` 等
```

### 7. 实施步骤

按时间顺序列出开发者应该执行的步骤（TDD 驱动）：

```
步骤 1：创建文件骨架
  - 创建 xxx.ts，定义接口，导出空实现
  - 创建 xxx.test.ts，编写第一批测试用例

步骤 2：实现核心功能 A（Red → Green）
  - 运行测试确认红灯
  - 实现功能 A 的最小代码
  - 运行测试确认绿灯

步骤 3：实现核心功能 B（Red → Green）
  ...

步骤 N：重构（Refactor）
  - 提取公共逻辑
  - 运行全部测试确认不回归
```

### 8. 验收标准

- 明确列出本阶段完成的判定条件
- 测试通过要求（单元测试 / 集成测试）
- 类型检查要求

## 编写流程

### 第一步：确认阶段

询问用户要为哪个阶段编写 plan。参考 docs/todo/ 下的任务清单获取阶段列表和任务描述。

### 第二步：研读架构设计文档（强制）

在编写 plan 前，必须阅读以下文档获取上下文：

- docs/architecture/ 下的所有相关设计文档 —— 这是 plan 的设计基础，不可跳过
- docs/todo/ 下的任务清单 —— 该阶段的任务清单和测试用例
- 项目 CLAUDE.md —— 项目工程规范
- 如果前置阶段已有 plan，阅读前置 plan 了解已有实现

### 第三步：生成编码并编写 plan

生成 6 位随机编码，按照上述文档结构逐章编写：

- 严格基于架构设计：所有接口、类、依赖关系必须与架构文档一致
- 具体到代码级别：接口定义给出完整的 TypeScript 代码，不是概念描述
- 可直接执行：开发者读完 plan 后无需再查阅其他文档即可开始编码
- 测试先行：实施步骤中测试编写必须在实现代码之前
- 遵循工程规范：路径别名用 @/，跨 package 用 workspace:*，全 strict TypeScript

### 第四步：确认写入

向用户确认 plan 内容无误后，写入 `docs/plan/{编码}-{计划名称}.md`

### 第五步：更新索引

在 `docs/plan/000000-index.md` 中添加新 plan 的条目（包含编码、文件名、内容说明、状态）。

## 注意事项

- 架构一致性是硬约束：Plan 中的接口设计必须与 docs/architecture/ 中的设计保持一致，如发现冲突需提示用户，等待用户决策后再继续
- Plan 是面向执行的文档，不要写空泛的描述，每一句都应该是可操作的
- 如果阶段内容过多，可以拆分为多个子 plan 文件，各自独立编码
